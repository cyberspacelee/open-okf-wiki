# Operator Session first, HITL publication, and framework-native interaction

The operator-facing product is an installable **Operator Session** shell (conversation TUI by default), not a one-shot JSON CLI as the primary UX. A **Wiki Run** remains the bounded production job (frozen Snapshot Set, Host validation, atomic publication, Wiki Run Record) and may be started from a Session; it is never a resumable Semantic Workflow or chat transcript. Publication after successful Host validation and Wiki Reviewer inspection requires human approval by default via Pydantic AI deferred tool approval (`requires_approval` / `HandleDeferredToolCalls`); YOLO or explicit non-interactive `--yes` maps to `approve_all` and must not disable mounts, mechanical validation, or publication locks. Declining publication leaves the Staging Wiki intact and does not change the Published Wiki. Wiki Run Record statuses distinguish published success from `awaiting_publication` and `publication_declined` (exact enum strings may match these names). Streaming, compaction, overflow, multi-turn history, step persistence, and sub-agent orchestration use Pydantic AI / pydantic-ai-harness capabilities rather than bespoke runtimes; the Host and Session shell stay thin adapters. Operator errors redact secrets but keep diagnosable messages (no blanket “provider diagnostics withheld” for missing credentials and similar operator-safe failures). Wiki Visualization remains a separate static artifact of the Published Wiki (ADR 0016).
