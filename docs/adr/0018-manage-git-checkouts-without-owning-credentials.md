# Manage git checkouts without owning credentials

A Workspace may clone and pull its configured Source Checkouts, but delegates authentication to the user's existing Git configuration, SSH agent, and credential helpers. A managed checkout is cloned under `<workspace>/sources/<source-id>/`; a linked checkout registers an existing local repository without moving, copying, or assuming ownership of it. Each Source Revision Policy either follows a named branch or pins an exact commit, but every Production Run resolves that policy to an immutable Source Snapshot revision. Pull is blocked when a checkout has uncommitted or untracked content; the Workspace never stashes, resets, cleans, or overwrites user changes, and dirty or untracked content never enters the authoritative Source Set.

Managed deletion defends against path escape, symlinks, mount traversal, ownership mismatch, and ordinary non-cooperating namespace races by using receipts, directory descriptors, mount identities, and quarantine names. It fails closed when mount ownership cannot be determined. An actively adversarial process running as the same operating-system user is outside this boundary: it already has equivalent authority over the user's files, and POSIX provides no atomic unlink-by-descriptor or mandatory directory-entry lock. Detected concurrent changes still abort deletion and retain the receipt for recovery.
