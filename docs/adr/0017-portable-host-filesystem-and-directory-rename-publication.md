# Use a portable Host filesystem policy and same-volume directory-rename publication

Wiki Run staging and publication remain Host-owned safety boundaries (see ADR 0006 and ADR 0007): the Agent still writes only into an isolated Staging Wiki, Python still validates mechanical invariants, and an incomplete or failed run must never leave a partial tree as the Published Wiki. This decision keeps those product promises and changes only how the Host implements path safety and atomic exposure.

The Host no longer requires Linux-only primitives (`dir_fd` with `O_NOFOLLOW` / `O_DIRECTORY`, or `/proc/self/fd` as a stable directory path) as a hard runtime gate for Wiki Run. Default path policy is portable: configured roots are absolute and non-overlapping; path components must not be symbolic links or other host reparse points the implementation can detect; single-file handoffs (Wiki Run Record, Analysis Receipts, publication metadata) use temporary file then `os.replace` on the same filesystem. The default threat model is a single-operator, Host-controlled workspace—not multi-tenant concurrent mutation of the publication path. A stricter openat-style backend may exist later as an explicit opt-in on platforms that support it; it is not the product baseline.

Publication no longer depends on a producer-managed symlink at the Published Wiki path. The Host materializes each successful release as a complete directory under a sibling releases root (same parent volume as the Published Wiki path), validates that tree, then exposes it by same-volume directory rename so readers see either the previous complete Wiki or the new complete Wiki—not a half-written tree. Staging, skill, snapshot, and publication paths must not overlap, and the releases root must share a volume with the Published Wiki path so rename stays atomic; cross-volume layouts fail closed at prepare time rather than falling back to copy. Concurrent Wiki Runs that target the same Published Wiki path fail closed (Host exclusive publication lock); they do not interleave renames. After a successful swap, the Host may remove superseded release directories that are no longer the live tree; retention of historical releases is not a product feature in this step. Git is not adopted as the publication store in this step.

Swap recovery follows best-effort restore, not a guarantee that the stable Published Wiki path is non-absent under every crash: if the Host has moved the previous tree aside and cannot install the new tree, it attempts to rename the previous tree back; if restore fails, it leaves recoverable paths (for example the aside tree and the validated release) and surfaces a clear operator error. The Host must never leave a half-written directory as the Published Wiki. A brief absence of the stable path under extreme interruption is acceptable when recovery artifacts remain diagnosable.

This deliberately rejects keeping Linux-only fd publication, defaulting to an insecure “plain pathlib with no symlink checks” mode, and adopting a git-ref backend solely to obtain atomicity. Platform support for `wiki-run` / publication is therefore not inherently limited to Linux; Windows (and other hosts that implement the portable policy and same-volume rename) are in scope for the Host filesystem. Existing deployments that treat the Published Wiki path as a symlink are a one-time layout migration concern; the migration policy is chosen explicitly when implementing the layout change.
