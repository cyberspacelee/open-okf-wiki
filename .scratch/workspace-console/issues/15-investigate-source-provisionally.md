# 15 — Investigate a Source Snapshot provisionally

**What to build:** When accepted knowledge cannot answer a question, a user can explicitly investigate the fixed Source Snapshot and receive a source-cited provisional result that cannot bypass normal knowledge production, verification, review, or publication.

**Blocked by:** 05 — Pull safely and resolve Source Revision Policies; 14 — Ask the Accepted Knowledge Model.

**Status:** ready-for-agent

- [ ] An insufficient Knowledge Query can offer a separate Investigate Source action without automatically switching authority modes.
- [ ] The investigation view clearly labels its scope and every result as provisional and not part of the Knowledge Bundle.
- [ ] A Source Investigation is bound to one fixed Run and Source Snapshot set, never a moving checkout or branch name.
- [ ] The investigation uses only bounded list, literal search, and read tools constrained to registered sources and canonical paths.
- [ ] Every factual result cites exact source ID, revision, path, span, and digest.
- [ ] Repository instructions and content cannot expand tool authority, enable shell execution, request credentials, or trigger Web Enrichment.
- [ ] Investigation results cannot create or change Claims, Concepts, Evidence References, Coverage Obligations, Findings, review decisions, Bundle pages, or publication state.
- [ ] The UI does not allow a provisional result to be accepted directly; authoritative adoption requires a later normal Production Run.
- [ ] Query and Source Investigation histories remain visually and semantically distinct.
- [ ] Audit records approved non-content metadata and citations without silently persisting investigation prose.
- [ ] Evaluations cover scope, citations, provisional labeling, prompt injection, refusal, budgets, cost, and attempts to mutate authority.
- [ ] End-to-end tests prove that a successful investigation leaves the authoritative digest, review snapshot, and published Bundle unchanged.
